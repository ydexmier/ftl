## Cas limites non testés dans les fichiers existants

### `auth-login.test.ts`

- **Refresh de session** : `POST /api/auth/refresh` (utilisé par `SessionGuard` toutes les 4 min) n'a aucun test. C'est la route qui renouvelle le sliding window du cookie ET de la session BDD. Absence de test = risque silencieux de déconnexion intempestive.
- **Cookie malformé** : aucun test ne vérifie le comportement quand le cookie est présent mais sa signature HMAC est invalide (doit retourner 401, pas 500).

### `conflictResolutionAdmin.test.ts`

- Résolution d'un conflit déjà `APPROVED` par l'admin (idempotence ou 400 ?).
- Aucun test pour `UNCERTAINTY` dans le workflow admin.

### `conflictResolutionMember.test.ts`

- Aucun test pour la transition `PENDING → UNCERTAINTY` avec `resolvedInks` déjà renseignés.
- Aucun test pour le cas où le joueur du conflit n'existe plus dans le tournoi.

### `groups-tournaments.test.ts`

- `GET /api/groups/[id]/tournaments/[tid]` (détail d'un tournoi de groupe) : aucun test, seuls PATCH et DELETE sont couverts.
- Accès externe avec un utilisateur déjà membre du groupe → attendu 409.

### `rounds.test.ts`

- `POST /api/rounds/[roundId]/matchs/[matchId]/assign_deck` avec `groupId` (scope groupe) : seul le scope utilisateur/tournoi est testé.
- Assignation d'un deck invalide (encre inexistante) → attendu 400.

### `dataMerge.test.ts`

- Aucun test pour le cas où plusieurs conflits sont créés pour le même joueur en une seule passe (doublon de conflit possible ?).
- Aucun test pour `mergeOnGroupJoin` quand l'utilisateur n'a aucun tournoi personnel.

### `tournaments.test.ts`

- `POST /api/tournaments` (liaison tournoi → utilisateur, hors admin) : aucun test.

## Liste priorisée des tests à écrire

### Lot 1 — 🔴 Critique

#### 1. `src/__tests__/auth-refresh.test.ts` — Test d'intégration

**Fichier cible :** `app/api/auth/refresh/route.ts`

Tester :
- `POST /api/auth/refresh` avec session valide → 200, cookie réémis avec une nouvelle expiration
- `POST /api/auth/refresh` sans cookie → 401
- `POST /api/auth/refresh` avec session expirée en BDD (TTL absolu dépassé) → 401
- `POST /api/auth/refresh` avec cookie dont la signature HMAC est invalide → 401
- Vérifier que `lastActivityAt` est mis à jour en BDD après un refresh réussi
- Vérifier que le `Set-Cookie` de la réponse porte bien une `maxAge` de 4h fraîche

> Cette route est le cœur du sliding window de session. Son absence de test est le seul risque 🔴 restant.

### Lot 2 — 🟠 Important

#### 2. `src/__tests__/auth-pure.test.ts` — Tests unitaires supplémentaires

**Fichier cible :** `src/lib/auth/cookieSign.ts`

Ajouter dans le fichier existant :
- `signCookie(sessionId, role)` → retourne une chaîne non vide en base64url
- `verifyCookie(signed)` → retourne `{ sessionId, role }` pour un cookie valide
- `verifyCookie(tampered)` → retourne `null` pour un cookie avec signature modifiée
- `verifyCookie('')` → retourne `null` (pas d'exception)
- Round-trip : `verifyCookie(await signCookie(id, role))` → retrouve id et role exacts

> `cookieSign.ts` est la primitive de sécurité fondamentale du middleware. Un bug ici bypasserait toute l'authentification.

#### 3. `src/__tests__/feedback.test.ts` — Test d'intégration
