## Liste priorisée des tests à écrire

### Lot 1 — 🔴 Critique (auth & données utilisateur)

#### 1. `src/__tests__/auth-login.test.ts` — Tests d'intégration
**Fichiers cibles :** `app/api/auth/login/route.ts`, `app/api/auth/logout/route.ts`, `app/api/auth/me/route.ts`

Tester :
- `POST /api/auth/login` → 200 + cookie signé avec identifiants valides
- `POST /api/auth/login` → 401 avec mot de passe incorrect
- `POST /api/auth/login` → 401 avec email inconnu (pas d'énumération)
- `POST /api/auth/login` → 429 après 5 tentatives échouées (rate limiting)
- `POST /api/auth/login` → 400 avec corps invalide (email manquant)
- `GET /api/auth/me` → 200 avec session valide, retourne les données utilisateur
- `GET /api/auth/me` → 401 sans session
- `POST /api/auth/logout` → 200, cookie supprimé, session détruite en BDD

#### 2. `src/__tests__/admin-users.test.ts` — Tests d'intégration
**Fichiers cibles :** `app/api/admin/users/route.ts`, `app/api/admin/users/[id]/route.ts`, `app/api/admin/users/[id]/sessions/route.ts`

Tester :
- `GET /api/admin/users` → 401 sans session, 401 rôle USER, 200 rôle ADMIN avec liste
- `GET /api/admin/users` → pagination et filtre de recherche
- `GET /api/admin/users/[id]` → 404 si ID inconnu, 200 avec données utilisateur
- `PATCH /api/admin/users/[id]` → changement de rôle par SUPERUSER uniquement
- `PATCH /api/admin/users/[id]` → blocage modification d'un autre SUPERUSER par ADMIN (RBAC)
- `DELETE /api/admin/users/[id]/sessions` → révocation de toutes les sessions d'un utilisateur

#### 3. `src/__tests__/auth-pure.test.ts` — Tests unitaires
**Fichiers cibles :** `src/lib/auth/password.ts`, `src/lib/auth/rbac.ts`

Tester :
- `validatePasswordStrength` : < 12 chars → invalid, sans majuscule → invalid, sans chiffre → invalid, sans spécial → invalid, mot de passe conforme → valid
- `hasRole('USER', 'ADMIN')` → false
- `hasRole('ADMIN', 'ADMIN')` → true
- `hasRole('SUPERUSER', 'ADMIN')` → true
- `hasRole('USER', 'USER')` → true
- Rôle inconnu → false (pas d'erreur)

#### 4. `src/__tests__/audit-logs.test.ts` — Tests d'intégration
**Fichier cible :** `app/api/admin/audit-logs/route.ts`

Tester :
- `GET /api/admin/audit-logs` → 401 sans session, 401 rôle USER, 200 rôle ADMIN
- Vérifier que les actions sensibles (login, changement de mot de passe) créent bien des entrées en audit log
- Pagination et filtrage par action/date

#### 6. `src/__tests__/groups-members.test.ts` — Tests d'intégration
**Fichiers cibles :** `app/api/groups/[id]/members/route.ts`, `app/api/groups/[id]/members/[userId]/route.ts`, `app/api/groups/invitations/route.ts`, `app/api/groups/invitations/[invId]/route.ts`

- Retrait d'un membre → 403 si non admin (sauf self-remove)
- Retrait du seul admin → 400 (règle métier GroupService)
- Changement de rôle → MEMBER → ADMIN et retour
