### ARCH-01 — Requêtes Mongoose directes dans les Server Components pages admin

**Contexte :**  
Les pages Server Components `app/admin/users/page.tsx`, `app/admin/users/[id]/page.tsx` et `app/admin/invitations/page.tsx` importent et requêtent directement les modèles Mongoose (`UserModel`, `SessionModel`, `InvitationModel`) sans passer par les repositories. C'est la même violation de principe que celle corrigée dans les routes API au sprint précédent, étendue à la couche présentation. Si les modèles ou leur schéma évoluent, ces pages sont des points de casse silencieux.

**Critères d'acceptation :**
- [ ] `app/admin/users/page.tsx` n'importe plus `UserModel` — utilise `UserRepository.findWithFilters()`
- [ ] `app/admin/users/[id]/page.tsx` n'importe plus `UserModel`, `SessionModel`, `AuditLogModel` — utilise les repositories correspondants
- [ ] `app/admin/invitations/page.tsx` n'importe plus `InvitationModel` — utilise `InvitationRepository.findWithFilters()` avec populate si nécessaire (enrichir le repository si besoin)
- [ ] Aucun appel `connectToMongoDB()` direct dans ces pages (géré par le repository)
- [ ] Les pages continuent de fonctionner (affichage correct des données)

**Effort estimé :** S  
**Dépendances :** Aucune (repositories existent déjà)  
**Source :** Rapport

### ARCH-03 — Types métier importés depuis les modèles Mongoose dans les routes API

**Contexte :**  
Trois routes API importent des types (`FeedbackStatus`, `UserTournamentStatus`) directement depuis les fichiers de modèles Mongoose (`@models/Feedback`, `@models/UserTournament`). Ces types devraient résider dans `src/types/` pour que la couche API ne soit pas couplée à la couche modèle. Ce couplage est mineur mais crée une dépendance indirecte qui se propage à tout fichier qui importe ces routes.

**Critères d'acceptation :**
- [ ] `FeedbackStatus` et `FeedbackType` sont définis dans `src/types/feedback.ts`
- [ ] `UserTournamentStatus` est défini dans `src/types/userTournament.ts` (ou `src/types/tournament.ts` si cohérent)
- [ ] Les modèles Mongoose importent ces types depuis `src/types/` (pas de duplication)
- [ ] Les routes `app/api/admin/feedback/route.ts`, `app/api/admin/feedback/[id]/route.ts`, `app/api/user/tournaments/[id]/route.ts` importent depuis `src/types/`
- [ ] Les pages admin (`app/admin/feedback/page.tsx`, `app/admin/audit-logs/page.tsx`) importent depuis `src/types/` également

**Effort estimé :** XS  
**Dépendances :** Aucune  
**Source :** Rapport

### REFACTO-01 — Standardiser les mutations HTTP dans les composants React

**Contexte :**  
20+ composants React font des appels `fetch()` directs pour les mutations (POST/PUT/PATCH/DELETE), chacun réimplémentant sa propre gestion d'erreur, état de chargement et headers. `src/lib/api/apiFetch.ts` existe pour les cas simples mais n'est pas utilisé comme standard. Cette dispersion rend les changements d'API (ex. ajout d'un header d'authentification, centralisation des messages d'erreur) difficiles à propager.

**Critères d'acceptation :**
- [ ] `src/lib/api/apiFetch.ts` est documenté comme le standard pour les appels API côté client (GET et mutations)
- [ ] Au minimum les composants les plus utilisés migrent vers `apiFetch` : `FeedbackWidget.tsx`, `OnboardingTour.tsx`, `DeleteTournamentButton.tsx`, `PlayerDeckModal.tsx`
- [ ] Les appels `fetch()` directs dans les composants migrés sont supprimés
- [ ] Le comportement fonctionnel est identique après migration
- [ ] Aucune régression dans les tests existants

**Effort estimé :** M  
**Dépendances :** Aucune  
**Source :** Rapport

### REFACTO-02 — Extraire CollapsibleSection et GroupSubSection de TournamentsPageClient

**Contexte :**  
`TournamentsPageClient.tsx` (365 lignes) définit encore 3 composants React dans un seul fichier. La logique d'état a été extraite dans un hook au sprint précédent, mais les composants présentationnels `CollapsibleSection` et `GroupSubSection` restent inline. Cela rend le fichier difficile à naviguer et empêche la réutilisation potentielle de ces composants.

**Critères d'acceptation :**
- [ ] `CollapsibleSection` est extrait dans `components/tournament/CollapsibleSection.tsx`
- [ ] `GroupSubSection` est extrait dans `components/tournament/GroupSubSection.tsx`
- [ ] `TournamentsPageClient.tsx` n'exporte que `TournamentsPageClient` et importe les deux composants depuis leurs fichiers
- [ ] Le comportement visuel est identique
- [ ] `TournamentsPageClient.tsx` fait moins de 200 lignes après extraction

**Effort estimé :** S  
**Dépendances :** Aucune  
**Source :** Rapport

### REFACTO-03 — Harmoniser le nommage des repositories

**Contexte :**  
`FeedbackRepository` expose `findAll()` tandis que tous les autres repositories avec listes paginées exposent `findWithFilters()`. Cette inconsistance oblige les développeurs à vérifier le nom exact dans chaque repository au lieu d'appliquer une convention.

**Critères d'acceptation :**
- [ ] `FeedbackRepository.findAll()` est renommé en `findWithFilters()`
- [ ] Tous les appelants de `FeedbackRepository.findAll()` sont mis à jour
- [ ] Les tests existants de feedback continuent de passer

**Effort estimé :** XS  
**Dépendances :** Aucune  
**Source :** Rapport

### TS-01 — Supprimer le `as any` résiduel dans DeleteTournamentButton

**Contexte :**  
`components/tournament/DeleteTournamentButton.tsx` ligne 26 utilise `(response as any).data` pour accéder au tournoi retourné par la route DELETE. La réponse n'est pas typée, forçant un cast qui masque les erreurs de type.

**Critères d'acceptation :**
- [ ] La réponse de la requête DELETE est typée avec une interface (`{ data: Tournament }` ou similaire)
- [ ] `as any` supprimé à la ligne 26
- [ ] TypeScript strict ne lève aucune nouvelle erreur
- [ ] Le comportement fonctionnel est identique

**Effort estimé :** XS  
**Dépendances :** Aucune  
**Source :** Rapport

### TEST-02 — Tests manquants pour GET /api/admin/users/[id]/sessions

**Contexte :**  
`app/api/admin/users/[id]/sessions/route.ts` dispose d'un test pour `DELETE` mais pas pour `GET` (liste des sessions actives d'un utilisateur). Cette méthode expose des données de session sensibles — une régression ou un bug de filtrage passerait sans alerte.

**Critères d'acceptation :**

- [ ] `GET /api/admin/users/[id]/sessions` avec session ADMIN → 200, liste des sessions actives
- [ ] `GET /api/admin/users/[id]/sessions` pour userId inconnu → réponse appropriée (liste vide ou 404)

**Effort estimé :** XS

**Source :** Rapport
