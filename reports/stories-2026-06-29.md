## SEC-01 — Révoquer les sessions actives lors d'un changement de rôle

**Catégorie :** Sécurité
**Priorité :** 🔴 Critique
**Effort :** S
**Dépendances :** aucune
**Source :** Rapport (identifiée le 06-22, **non corrigée depuis 3 semaines** — bloquée par la non-fusion des PR #202/#203)

### Contexte

Le rôle d'un utilisateur est figé dans son cookie signé au moment du login (`signCookie(sessionId, role)`) et n'est jamais relu depuis le document `User` à jour — ni dans `middleware.ts` (protection des pages `/admin/**`), ni dans `getAuthSession()` (protection des routes API). Quand un admin rétrograde un utilisateur via `PATCH /api/admin/users/[id]`, ses sessions existantes ne sont pas révoquées : il conserve ses anciens privilèges jusqu'à expiration de sa session (jusqu'à 12h) ou révocation manuelle via `DELETE /api/admin/users/[id]/sessions`. C'est un défaut de conception RBAC qui rend une rétrogradation de rôle inopérante en pratique pendant plusieurs heures. Revérifié ligne par ligne le 06-29 : toujours présent, code strictement identique.

### Critères d'acceptation

- [ ] `PATCH /api/admin/users/[id]` (`app/api/admin/users/[id]/route.ts`) détecte si `role !== undefined && role !== user.role`
- [ ] Si le rôle change, `SessionRepository.deleteByUserId(id)` est appelé après la mise à jour réussie de l'utilisateur
- [ ] Un test d'intégration vérifie : créer une session ADMIN active, la rétrograder en USER via l'API, vérifier que l'ancien cookie ne donne plus accès à une route `requireAdminSession` (401 ou 403 selon le comportement de session expirée)
- [ ] Un test d'intégration vérifie qu'une mise à jour de `username`/`email`/`canCreateGroup` (sans changement de `role`) ne révoque PAS les sessions actives (pas de régression sur le comportement existant)
- [ ] `npm test` passe à 100%

## SEC-02 — Exclure `sessionId` brut de la réponse `/api/admin/users/[id]/sessions`

**Catégorie :** Sécurité
**Priorité :** 🟠 Important
**Effort :** XS
**Dépendances :** aucune
**Source :** Rapport (identifiée le 06-22, non corrigée — 3 semaines)

### Contexte

`SessionRepository.findByUserId` (`src/repositories/db/SessionRepository.ts:10-13`) retourne les documents `Session` complets sans projection, incluant le champ `sessionId` — l'identifiant de session brut. Il est ensuite renvoyé tel quel par `GET /api/admin/users/[id]/sessions`. Ce champ n'est pas affiché dans l'UI admin (qui montre `ipAddress`, `userAgent`, `lastActivityAt`) et constitue une exposition de données inutile (principe de minimisation), même si une prise de contrôle de session directe n'est pas possible sans connaître `SESSION_SECRET`.

### Critères d'acceptation

- [ ] `SessionRepository.findByUserId` exclut le champ `sessionId` via `.select('-sessionId')`
- [ ] La réponse JSON de `GET /api/admin/users/[id]/sessions` ne contient plus le champ `sessionId`
- [ ] Un test d'intégration vérifie que la réponse de cette route ne contient pas la clé `sessionId` dans les objets `sessions[]`
- [ ] `npm test` passe à 100%

## SEC-03 — Sécuriser ou supprimer `POST /api/rounds/[roundId]/matchs`

**Catégorie :** Sécurité
**Priorité :** 🟠 Important
**Effort :** XS
**Dépendances :** aucune
**Source :** Rapport (nouvelle découverte du 06-29)

### Contexte

`app/api/rounds/[roundId]/matchs/route.ts:36-44` expose un `POST` sans aucune vérification de session ni validation du corps de requête, contrairement au `GET` du même fichier qui appelle `getAuthSession`. Le body est spreadé directement dans `RoundRepository.upsert({ id: Number(roundId), ...body })`. Le middleware bloque les requêtes sans cookie de session valide (redirection `/login`), mais n'importe quel utilisateur authentifié de n'importe quel rôle — y compris un compte invité `isGuest: true` — peut écraser un document `Round` arbitraire avec des données non validées, en contournant `RoundService.fetchAndSave` (la voie normale de récupération depuis Ravensburger).

Recherche d'usage effectuée (`useRound`, `useDeckAssignment`, `Match.tsx`, tests existants) : aucun appelant frontend ni test ne référence ce `POST`. C'est une route orpheline, probablement un reliquat antérieur à l'introduction de `RoundService.fetchAndSave` via `/api/rounds/fetch`.

### Critères d'acceptation

- [ ] Décision explicite prise et documentée : suppression de la route si confirmée inutilisée, ou sécurisation si un usage légitime est identifié
- [ ] Si suppression : le handler `POST` est retiré de `app/api/rounds/[roundId]/matchs/route.ts`, et un test confirme que `POST` sur cette route retourne 405 (ou n'existe plus)
- [ ] Si conservation : `requireAdminSession` est ajouté en début de handler (cohérent avec les autres points d'entrée de fetch Ravensburger) et le body est validé champ par champ avant d'être transmis à `RoundRepository.upsert`
- [ ] Un test d'intégration couvre le comportement choisi (rejet 401/403/405 selon le cas)
- [ ] `npm test` passe à 100%
