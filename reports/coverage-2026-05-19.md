## 4. Hooks React

**Note :** Les hooks React côté client ne peuvent pas être testés avec Vitest sans jsdom + React Testing Library. Leur absence de tests est un **angle mort distinct** des tests d'intégration des routes API.

| Hook | Complexité | Statut | Commentaire |
|------|-----------|--------|-------------|
| `useFetch<T>(url)` | Faible | ❌ Non testé | Hook générique GET, logique simple |
| `useTournament` | Moyenne | ❌ Non testé | Wraps useFetch + refreshTournament() |
| `useRound` | Élevée | ❌ Non testé | Matches paginés, mutations decks, logique d'état non triviale |
| `useDeckAssignment` | Moyenne | ❌ Non testé | État formulaire d'assignation |
| `useMatchState` | Élevée | ❌ Non testé | Reducer de transition d'état des decks (extrait de MatchModal) |
| `useDebounce` | Faible | ❌ Non testé | Trivial, non prioritaire |

## 7. Tests prioritaires à écrire

### Lot 1 🔴 — Priorité haute (blocage qualité)

#### `/api/tournaments/[id]/last-round`
Fichier cible : `src/__tests__/last-round.test.ts`

Cas à couvrir :
- [ ] `GET` sans cookie → 401
- [ ] `GET` avec ID tournoi invalide (non-numérique) → 400
- [ ] `GET` pour un tournoi sans ronde fetchée → `{ roundId: null }`
- [ ] `GET` pour un tournoi avec plusieurs rondes → retourne l'ID de la dernière ronde par `lastFetchedAt`

#### `/api/tournaments/[id]/comment-counts`
Fichier cible : `src/__tests__/comment-counts.test.ts`

Cas à couvrir :
- [ ] `GET` sans cookie → 401
- [ ] `GET` avec `playerIds` vide → `{ counts: {} }`
- [ ] `GET` avec `playerIds` valides → retourne les counts corrects par playerId
- [ ] `GET` avec `groupId` pour non-membre → 403
- [ ] `GET` avec `groupId` pour un utilisateur avec accès externe actif → 200
- [ ] `GET` en portée personnelle (sans groupId) → counts de la portée user

### Lot 2 🟠 — Priorité moyenne

#### Hooks React — `useRound` et `useMatchState`
Ces hooks ont une logique d'état non triviale. Ils nécessitent la mise en place de `jsdom` + `React Testing Library` dans Vitest pour être testés. Cet investissement technique est justifié par l'importance de `useRound` dans le flux de scouting.

Fichier cible : `src/__tests__/hooks/useRound.test.ts`, `src/__tests__/hooks/useMatchState.test.ts`

Cas `useRound` :
- [ ] Chargement initial des matches avec decks
- [ ] Mutation optimiste lors d'une assignation de deck
- [ ] Rollback en cas d'erreur API
- [ ] Pagination (chargement page suivante)

Cas `useMatchState` :
- [ ] RESET remet l'état à l'état initial
- [ ] SELECT_DECK met à jour les decks de la bonne combinaison
- [ ] ASSIGN_PLAYER assigne le bon joueur et l'autre joueur à l'autre combinaison
- [ ] SET_INIT initialise l'état depuis les decks existants

#### `/api/groups/external-access/[accessId]` — couverture complète
Fichier cible : enrichir `groups-members.test.ts` ou créer `external-access.test.ts`

Cas à couvrir :
- [ ] Acceptation d'un accès externe
- [ ] Refus d'un accès externe
- [ ] Accès expiré → 404 ou 400

### Lot 3 🟡 — Amélioration continue

#### `TournamentExternalAccessRepository` — couverture directe
Fichier cible : enrichir les tests existants

Cas à couvrir :
- [ ] `hasActiveAccess()` avec accès ACCEPTED non expiré → true

#### Routes `search`, `link`, `my-role`
Ces routes sont implicitement couvertes mais méritent des tests dédiés pour les cas limites.
