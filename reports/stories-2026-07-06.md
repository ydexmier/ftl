### SEC-01 — Masquer les messages d'erreur internes dans `serverError()`

**Source :** Rapport (audit-dette-2026-07-06.md §1.1)  
**Priorité :** 🔴 Critique  
**Effort :** XS

**Contexte :**  
`ApiResponse.serverError(err)` renvoie le message brut de l'exception au client HTTP. En production, cela expose des détails internes de Mongoose : nom de la base de données, collection, contraintes d'index (ex. `MongoServerError: E11000 duplicate key error collection: ftl.users index: username_1`). Cette fonction est appelée dans ~50 routes.

**Critères d'acceptation :**
- [ ] `ApiResponse.serverError()` renvoie toujours `{ error: "Erreur interne du serveur." }` en réponse HTTP
- [ ] Le `console.error(err)` côté serveur est conservé (log interne)
- [ ] Les routes qui passent un message personnalisé utilisent un paramètre optionnel `message?: string`
- [ ] Aucun test existant ne casse suite à la modification

**Dépendances :** Aucune

### SEC-02 — Rate limiting sur `GET /api/invitations/[token]`

**Source :** Rapport (audit-dette-2026-07-06.md §1.2)  
**Priorité :** 🟠 Important  
**Effort :** XS

**Contexte :**  
Le handler GET de `/api/invitations/[token]` est public (sans authentification), retourne l'email associé au token, et n'a aucun rate limiting par IP. Cela permet théoriquement de bruteforcer des tokens d'invitation pour découvrir des emails. Le risque est limité (UUIDs v4), mais le pattern est incohérent avec tous les autres endpoints similaires du projet.

**Critères d'acceptation :**
- [ ] `checkRateLimit('invite-get:${ip}')` est appelé au début du handler GET
- [ ] La limite est cohérente avec les autres endpoints (5 req/15 min par IP)
- [ ] Le handler retourne 429 avec message clair si la limite est dépassée
- [ ] Les tests dans `register.test.ts` passent sans modification

**Dépendances :** Aucune

- [ ] Les tests de `api-tokens.test.ts` passent sans modification
- [ ] Grep sur `import connectToMongoDB` dans `app/api/` ne retourne aucun résultat hors `test/seed/`

### ARCH-02 — Refactoriser les routes de seed E2E vers les repositories

**Source :** Rapport (audit-dette-2026-07-06.md §2.1)  
**Priorité :** 🟠 Important  
**Effort :** M

**Contexte :**  
`app/api/test/seed/route.ts` et `app/api/test/seed/scenarios/route.ts` accèdent directement aux modèles Mongoose (`UserModel.create()`, `GroupModel.deleteMany()`, etc.) en dehors du pattern repository. Ces routes sont bloquées en production mais la violation architecturale crée un précédent et empêche la réutilisation des contraintes définies dans les repositories (validation, normalisation).

**Critères d'acceptation :**
- [ ] Toutes les opérations Mongoose dans `seed/route.ts` passent par les repositories correspondants (UserRepository, GroupRepository, TournamentRepository, RoundRepository)
- [ ] Toutes les opérations dans `seed/scenarios/route.ts` passent par les repositories correspondants (GroupTournamentRepository, TournamentPlayersDeckRepository, TournamentConflictRepository)
- [ ] Des helpers `createTestUser()`, `createTestGroup()` etc. sont créés dans `src/test/helpers.ts` si utile pour la lisibilité
- [ ] Les tests E2E (Playwright) continuent de fonctionner après le refactoring

**Dépendances :** Aucune

## Stories de validation (VALID)

---

### VALID-01 — Valider le format email dans `external-access`

**Source :** Rapport (audit-dette-2026-07-06.md §3.1)  
**Priorité :** 🟡 Mineur  
**Effort :** XS

**Contexte :**  
La route `POST /api/groups/[id]/tournaments/[tid]/external-access` vérifie uniquement la présence du champ `email` (pas son format). Cela permet d'envoyer un magic link à une chaîne arbitraire non-email, ce qui provoque une erreur silencieuse côté Resend. La fonction `isValidEmail()` de `src/lib/validation.ts` est disponible mais non appelée ici.

**Critères d'acceptation :**
- [ ] `isValidEmail(email)` est appelé après la vérification de présence
- [ ] Le handler retourne 400 `'Email invalide.'` si le format ne correspond pas
- [ ] Un test dans `groups-tournaments.test.ts` couvre le cas email malformé
- [ ] Un test dans `groups-tournaments.test.ts` couvre le cas email absent

**Dépendances :** Aucune
