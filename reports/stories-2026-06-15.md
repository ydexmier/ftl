## ARCH-01 — Supprimer `connectToMongoDB()` redondant dans la révocation de token

**Catégorie :** Architecture  
**Priorité :** 🟠 Important  
**Effort :** XS  
**Dépendances :** aucune  
**Source :** Rapport

### Contexte

`app/api/groups/[id]/tournaments/[tid]/api-tokens/[tokenId]/route.ts` (lignes 6 et 18) appelle encore `connectToMongoDB()` directement avant d'invoquer `GroupRepository.isAdmin` et `ApiTokenRepository.revoke`. Ces repositories gèrent leur propre connexion en interne via le singleton `src/lib/db.ts`. La même violation a été corrigée la semaine dernière dans les deux autres routes `api-tokens`, mais cette route de révocation a été oubliée.

### Critères d'acceptation

- [ ] `import connectToMongoDB from '@/src/lib/db'` est supprimé du fichier
- [ ] `await connectToMongoDB()` (ligne 18) est supprimé
- [ ] Le test `api-tokens.test.ts` — cas "DELETE /api/groups/[id]/tournaments/[tid]/api-tokens/[tokenId]" — passe sans modification
- [ ] Aucun autre fichier de route (hors `test/seed*`) n'appelle `connectToMongoDB()` (`grep -r "connectToMongoDB" app/api/ --include="route.ts"` retourne seulement les deux routes de seed)

## TEST-01 — Tests pour `/api/guest/my-tournaments`

**Catégorie :** Couverture tests  
**Priorité :** 🔴 Lot 1  
**Effort :** S  
**Dépendances :** aucune  
**Source :** Rapport

### Contexte

La route `GET /api/guest/my-tournaments` est le point d'entrée des invités après leur connexion : elle liste les tournois auxquels ils ont accès. Elle n'a aucun test d'intégration. La logique de filtrage (accès ACCEPTED vs REVOKED vs EXPIRED) est non triviale et mérite d'être vérifiée.

### Critères d'acceptation

- [ ] Un fichier `src/__tests__/guest-my-tournaments.test.ts` existe (ou les cas sont ajoutés à `guest-access.test.ts`)
- [ ] Cas : `GET` sans session → 401
- [ ] Cas : `GET` avec utilisateur non-invité → 200 `{ tournaments: [] }`
- [ ] Cas : `GET` avec invité ayant un accès ACCEPTED → 200 contenant le tournoi lié (id, name, groupId, accessId)
- [ ] Cas : `GET` avec invité ayant un accès REVOKED → 200 `{ tournaments: [] }` (accès filtré)
- [ ] Cas : invité avec deux accès à deux tournois distincts → les deux sont retournés

## TEST-02 — Tests pour `/api/groups/[id]/tournaments/[tid]/magic-link`

**Catégorie :** Couverture tests  
**Priorité :** 🔴 Lot 1  
**Effort :** S  
**Dépendances :** aucune  
**Source :** Rapport

### Contexte

La route GET/POST `magic-link` génère le lien d'invitation invité. Elle est sans tests alors qu'elle contient de la logique d'accès (admin uniquement) et d'upsert (désactive l'ancien lien, crée un nouveau token UUID). Un bug ici aurait un impact direct sur la fonctionnalité guest access.

### Critères d'acceptation

- [ ] Un fichier de test couvre cette route (nouveau fichier ou ajout à `groups-tournaments.test.ts`)
- [ ] Cas GET : sans session → 401
- [ ] Cas GET : membre non-admin → 403
- [ ] Cas GET : admin, aucun lien existant → 200 `{ link: null }`
- [ ] Cas GET : admin, lien actif → 200 avec le lien (token, isActive: true)
- [ ] Cas POST : sans session → 401
- [ ] Cas POST : membre non-admin → 403
- [ ] Cas POST : tournoi non rattaché au groupe → 404
- [ ] Cas POST : premier appel → 201 avec lien créé (token non nul, isActive: true)
- [ ] Cas POST : second appel → 201 avec nouveau token, l'ancien est désactivé (isActive: false)

## TEST-03 — Tests pour routes de navigation groupe/tournoi

**Catégorie :** Couverture tests  
**Priorité :** 🟠 Lot 2  
**Effort :** S  
**Dépendances :** aucune  
**Source :** Rapport

### Contexte

Trois routes retournent des données de navigation/affichage pour l'UI :
- `GET /api/tournaments/[id]/admin-groups` — groupes du tournoi où l'user est admin
- `GET /api/tournaments/[id]/member-groups` — groupes du tournoi où l'user est membre
- `GET /api/groups/[id]/tournaments/[tid]/guests` — liste des accès externes au tournoi

Ces routes filtrent les données selon le rôle de l'utilisateur et n'ont aucun test.

### Critères d'acceptation

- [ ] `admin-groups` — 401 sans session
- [ ] `admin-groups` — 200 `{ groups: [] }` si l'user n'est admin d'aucun groupe lié au tournoi
- [ ] `admin-groups` — 200 avec les groupes où l'user est admin (filtre correctement les groupes où l'user est seulement membre)
- [ ] `member-groups` — 401 sans session
- [ ] `member-groups` — 200 avec les groupes membres du tournoi
- [ ] `guests` — 401 sans session
- [ ] `guests` — 403 si non-admin du groupe
- [ ] `guests` — 200 avec liste des accès externes (PENDING, ACCEPTED, REVOKED)

## REFACTO-01 — Extraire `useTournamentContext` de `Tournament.tsx`

**Catégorie :** Refactoring / God component  
**Priorité :** 🟡 Mineur  
**Effort :** M  
**Dépendances :** aucune  
**Source :** Rapport

### Contexte

`components/tournament/Tournament.tsx` atteint 398 lignes (+22 en une semaine) et contient un bloc `useEffect` de ~80 lignes avec 9 appels `fetch()` directs. - [ ] `Tournament.tsx` est réduit sous 320 lignes
- [ ] `Tournament.tsx` ne contient plus de `fetch()` directs dans des `useEffect` (seulement des appels aux hooks)
- [ ] `useTournamentContext` est testé dans `src/__tests__/hooks/useTournamentContext.test.ts` (cas : données retournées, état de chargement, rechargement)
- [ ] Pas de régression fonctionnelle sur la page tournoi

## TS-01 — Typer les pipelines MongoDB avec `PipelineStage`

**Catégorie :** Dette TypeScript  
**Priorité :** 🟡 Mineur  
**Effort :** S  
**Dépendances :** aucune  
**Source :** Rapport

### Contexte

7 usages de `any` subsistent dans `RoundRepository.ts` (4 occurrences) et `TournamentPlayersDeckRepository.ts` (3 occurrences) pour typer des tableaux de pipeline MongoDB. Ils sont fonctionnellement corrects et tous explicitement désactivés avec `eslint-disable-next-line`. Le type `PipelineStage` exporté par Mongoose v7 permettrait de les supprimer.

### Critères d'acceptation

- [ ] `import type { PipelineStage } from 'mongoose'` est ajouté dans les deux fichiers
- [ ] Les `const pipeline: any[]` sont remplacés par `const pipeline: PipelineStage[]`
- [ ] Les `const searchStages: any[]` et `const scoutingFilterStages: any[]` (RoundRepository) sont typés `PipelineStage[]`
- [ ] Le `as any` sur `$push` dans `TournamentPlayersDeckRepository` est remplacé par un cast approprié ou un type partiel de `PipelineStage`
- [ ] Les directives `eslint-disable-next-line @typescript-eslint/no-explicit-any` sont supprimées
- [ ] `npm test` passe à 100%
