### SEC-01 — Rate limiting sur `/api/auth/forgot-password`

**Contexte :** L'endpoint de demande de réinitialisation de mot de passe (`POST /api/auth/forgot-password`) n'a aucune limite de débit par IP. Un attaquant peut appeler cet endpoint en boucle avec l'email d'une victime pour inonder sa boîte mail, invalider les tokens précédents en chaîne, et générer des coûts d'envoi email en production (Resend facturation à l'usage).

**Critères d'acceptation :**
- [ ] Le handler POST applique `checkRateLimit('forgot-password:${ip}')` avant tout traitement
- [ ] Une IP bloquée reçoit 429 avec un message clair
- [ ] Les tests dans `password-reset.test.ts` couvrent le cas 429

**Effort :** XS  
**Dépendances :** Aucune  
**Source :** Rapport

### SEC-02 — Rate limiting sur `/api/auth/reset-password/[token]`

**Contexte :** Le handler POST de réinitialisation de mot de passe n'est pas limité en débit. Bien que le token soit un UUID cryptographique (difficile à brute-forcer), l'absence de limite facilite les attaques par timing et laisse la surface d'attaque ouverte si le schéma de tokens évolue.

**Critères d'acceptation :**
- [ ] Le handler POST applique `checkRateLimit('reset-password:${ip}')` avant tout traitement
- [ ] Une IP bloquée reçoit 429
- [ ] Test couvrant le cas 429 dans `password-reset.test.ts`

**Effort :** XS  
**Dépendances :** SEC-01 (pattern identique, préférable de faire ensemble)  
**Source :** Rapport

### SEC-03 — Rate limiting sur `/api/invitations/[token]`

**Critères d'acceptation :**
- [ ] Le handler POST applique `checkRateLimit('register-invite:${ip}')` avant tout traitement
- [ ] Une IP bloquée reçoit 429
- [ ] Test couvrant le cas 429 dans `register.test.ts`

**Effort :** XS  
**Dépendances :** SEC-01  
**Source :** Rapport

### ARCH-01 — Corriger le bug `(response as any).data` dans `DeleteTournamentButton.tsx`

**Contexte :** `components/tournament/DeleteTournamentButton.tsx` ligne 26 appelle `onDeleted?.((response as any).data)` où `response` est un objet `Response` de l'API Fetch. Ce type n'a pas de propriété `.data`, donc le cast `as any` masque le fait que le callback reçoit toujours `undefined` au lieu des données de la réponse JSON.

**Critères d'acceptation :**
- [ ] `response.json()` est appelé pour extraire les données
- [ ] `onDeleted` est appelé avec les données JSON réelles de la réponse
- [ ] Le cas d'erreur HTTP (non-ok) est géré : message d'erreur affiché à l'utilisateur
- [ ] Aucun `as any` dans le fichier

**Effort :** XS  
**Dépendances :** Aucune  
**Source :** Rapport

### VALID-01 — Schémas de validation déclaratifs dans les routes admin

**Contexte :** Les routes admin parsent `request.json()` et valident les champs de manière ad-hoc (checks manuels), à l'opposé des routes publiques et auth qui utilisent des fonctions centralisées dans `src/lib/validation.ts`. Cela crée une incohérence de maintenance et des risques d'oubli de validation sur certains champs (ex. format email dans `admin/users`, longueur du nom de groupe, validation des enums de statut/rôle).

**Routes concernées :**
- `app/api/admin/users/route.ts` — validation email, force password
- `app/api/admin/invitations/route.ts` — validation tableau d'emails
- `app/api/admin/feedback/[id]/route.ts` — validation enum `FeedbackStatus`
- `app/api/admin/groups/route.ts` POST — longueur nom
- `app/api/admin/groups/[id]/route.ts` PATCH — longueur nom
- `app/api/admin/groups/[id]/members/[uid]/route.ts` PATCH — enum rôle

**Critères d'acceptation :**
- [ ] `src/lib/validation.ts` contient des fonctions `validateAdminUserCreate`, `validateAdminUserUpdate`, `validateAdminGroupBody`, `validateAdminFeedbackStatus`, `validateAdminInvitationEmails`
- [ ] Chaque route admin utilise la fonction correspondante
- [ ] Les fonctions de validation retournent le type `ValidationResult<T>` existant
- [ ] Tests unitaires pour les nouvelles fonctions de validation (cas valides et invalides)

**Effort :** M  
**Dépendances :** Aucune  
**Source :** Rapport

### TEST-01 — Tests pour `/api/tournaments/[id]/last-round`

**Contexte :** Cette route, ajoutée avec la feature d'auto-sélection de la dernière ronde, n'a aucun test. Elle est utilisée par `Tournament.tsx` au chargement. Une régression (ex. mauvaise requête MongoDB, mauvaise sérialisation) passerait silencieusement.

**Critères d'acceptation :**
- [ ] Fichier `src/__tests__/last-round.test.ts` créé
- [ ] `GET` sans cookie → 401
- [ ] `GET` avec ID tournoi non-numérique → 400
- [ ] `GET` pour un tournoi sans ronde fetchée → `{ roundId: null }`
- [ ] `GET` pour un tournoi avec plusieurs rondes → retourne l'ID de la ronde avec le `lastFetchedAt` le plus récent

**Effort :** S  
**Dépendances :** Aucune  
**Source :** Rapport

### TEST-02 — Tests pour `/api/tournaments/[id]/comment-counts`

**Contexte :** Cette route agrège les counts de commentaires par joueur pour afficher des badges dans la vue ronde. Elle n'a aucun test. La logique d'autorisation (membre du groupe, accès externe, portée personnelle vs groupe) est non triviale.

**Critères d'acceptation :**
- [ ] Fichier `src/__tests__/comment-counts.test.ts` créé
- [ ] `GET` sans cookie → 401
- [ ] `GET` avec `playerIds` vide ou absent → `{ counts: {} }`
- [ ] `GET` avec `groupId` pour non-membre sans accès externe → 403
- [ ] `GET` avec `groupId` pour utilisateur avec accès externe actif → 200 avec counts
- [ ] `GET` en portée personnelle (sans groupId) → counts de la portée userId
- [ ] `GET` avec plusieurs `playerIds` → dict correct avec 0 pour les joueurs sans commentaires

**Effort :** S  
**Dépendances :** Aucune  
**Source :** Rapport

### TEST-03 — Tests de hooks React (`useRound`, `useMatchState`)

**Contexte :** `useRound` et `useMatchState` contiennent une logique d'état non triviale (mutations optimistes, reducer de transitions de deck) qui ne peut être vérifiée que par des tests de composant/hook. Leur absence de test est un angle mort identifié depuis l'audit 2026-05-11. La mise en place de React Testing Library dans Vitest est un prérequis.

**Critères d'acceptation :**
- [ ] `@testing-library/react` et `@testing-library/react-hooks` (ou équivalent Vitest) installés et configurés
- [ ] `vitest.config.ts` configuré avec `environment: 'jsdom'` pour les tests de hooks
- [ ] `src/__tests__/hooks/useMatchState.test.ts` avec cas : RESET, SELECT_DECK, ASSIGN_PLAYER, SET_INIT
- [ ] `src/__tests__/hooks/useRound.test.ts` avec cas : chargement initial, mutation optimiste, rollback erreur

**Effort :** L  
**Dépendances :** Aucune (mais setup infrastructure requis)  
**Source :** Rapport
