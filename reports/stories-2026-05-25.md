## SEC-01 — Ajouter l'authentification manquante sur 3 routes admin

**Identifiant :** SEC-01  
**Priorité :** 🔴 Critique  
**Effort :** XS  
**Source :** Rapport couverture (section 12 — Alerte sécurité)

### Contexte

Trois routes admin n'effectuent **aucun contrôle d'authentification** : elles ne font appel ni à `getAuthSession`, ni à `getAdminSession`, ni à aucune vérification de rôle. Le middleware Next.js vérifie le rôle ADMIN uniquement pour les routes UI `/admin/**` — les routes API `/api/admin/**` ne bénéficient que de la vérification HMAC du cookie. Tout utilisateur avec un cookie de session valide (rôle USER) peut donc appeler directement ces endpoints.

| Route | Impact |
|-------|--------|
| `POST /api/admin/fetchTournament` | N'importe quel USER peut déclencher un fetch Ravensburger |
| `POST /api/admin/fetchRound` | N'importe quel USER peut déclencher un fetch de ronde |
| `DELETE /api/admin/tournaments` | N'importe quel USER peut supprimer un tournoi en cascade |

### Critères d'acceptation

- [ ] `app/api/admin/fetchTournament/route.ts` : ajouter `getAuthSession` + vérification rôle ADMIN, retourner 401/403 si absent ou insuffisant
- [ ] `app/api/admin/fetchRound/route.ts` : idem
- [ ] `app/api/admin/tournaments/route.ts` : idem
- [ ] Tests dans `tournaments.test.ts` : `POST /api/admin/fetchTournament` sans cookie → 401, avec rôle USER → 403
- [ ] Tests dans `rounds.test.ts` : `POST /api/admin/fetchRound` sans cookie → 401, avec rôle USER → 403
- [ ] Tests dans `tournaments.test.ts` : `DELETE /api/admin/tournaments` sans cookie → 401, avec rôle USER → 403

**Dépendances :** Peut être fait avant ARCH-01 ; sera absorbé proprement une fois ARCH-01 terminé

## ARCH-01 — Unifier les patterns d'authentification admin

**Identifiant :** ARCH-01  
**Priorité :** 🟠 Important  
**Effort :** M  
**Source :** Rapport

### Contexte

Trois patterns d'authentification coexistent dans les routes admin, issus de l'accumulation de développements sans harmonisation :

1. `getAdminSession()` → retourne `{ parsed, session }`, appelle `connectToMongoDB()` directement (10 routes)
2. `getAuthSession()` + `hasRole()` inline (2 routes : `admin/feedback`, `admin/feedback/[id]`)
3. `getAdminAuth()` locale dans chaque fichier, retournant `{ auth, error }` (3 groupes de fichiers : `admin/groups/`, `admin/groups/[id]/members/`, `admin/groups/[id]/tournaments/`)

`getAdminSession` n'est pas documenté dans CLAUDE.md, retourne une interface différente de `getAuthSession` (`auth.session.userId` au lieu de `auth.userId`), et contient un appel `connectToMongoDB()` direct qui viole la séparation repository/route.

### Critères d'acceptation

- [ ] Supprimer `src/lib/auth/getAdminSession.ts`
- [ ] Créer `requireAdminSession(request: NextRequest)` dans `src/lib/auth/getAuthSession.ts` (ou fichier dédié documenté dans CLAUDE.md), s'appuyant sur `getAuthSession()` et `hasRole()`
- [ ] `requireAdminSession` retourne soit `{ session: AuthSession }` soit `{ error: NextResponse }` — même type `AuthSession` que `getAuthSession` (`{ userId, role, sessionId }`)
- [ ] Migrer les 10 routes utilisant `getAdminSession` vers `requireAdminSession`
- [ ] Supprimer les 3 fonctions `getAdminAuth()` locales et migrer vers `requireAdminSession`
- [ ] Migrer les 2 routes utilisant `getAuthSession + hasRole` inline vers `requireAdminSession`
- [ ] Aucun appel `connectToMongoDB()` hors des repositories et de la couche `src/lib/auth/session.ts`
- [ ] Tous les tests existants passent sans modification (comportement identique)
- [ ] `requireAdminSession` documenté dans CLAUDE.md section "Routes API"

**Dépendances :** Aucune

## ARCH-02 — Supprimer `connectToMongoDB()` redondant dans les routes admin

**Identifiant :** ARCH-02  
**Priorité :** 🟡 Mineur  
**Effort :** XS  
**Source :** Rapport

### Contexte

`app/api/admin/fetchRound/route.ts` et `app/api/admin/fetchTournament/route.ts` appellent `connectToMongoDB()` directement avant d'appeler un repository. Chaque repository gère sa propre connexion en interne via le singleton, rendant ces appels redondants et créant un couplage inutile entre la route et les détails d'implémentation DB.

### Critères d'acceptation

- [ ] Supprimer `import connectToMongoDB from '@/src/lib/db'` de `fetchRound/route.ts`
- [ ] Supprimer `await connectToMongoDB()` de `fetchRound/route.ts`
- [ ] Supprimer `import connectToMongoDB from '@/src/lib/db'` de `fetchTournament/route.ts`
- [ ] Supprimer `await connectToMongoDB()` de `fetchTournament/route.ts`
- [ ] Les tests existants passent sans modification

**Dépendances :** Aucune

## REFACTO-01 — Extraire les builders ECharts de `StatsTab.tsx`

**Identifiant :** REFACTO-01  
**Priorité :** 🟡 Mineur  
**Effort :** S  
**Source :** Rapport

### Contexte

`components/tournament/StatsTab.tsx` fait 433 lignes, dépassant le seuil des 300. Les 3 fonctions de construction d'options ECharts (`buildDonutOption`, `buildBarOption`, `buildHeatmapOption`) représentent ~270 lignes de logique de présentation, extractible en fonctions pures indépendantes. - [ ] Comportement visuel identique à avant
- [ ] Aucun test existant cassé

## TEST-01 — Tests pour `/api/tournaments/[id]/stats`

**Identifiant :** TEST-01  
**Priorité :** 🔴 Lot 1  
**Effort :** S  
**Source :** Rapport

### Contexte

`/api/tournaments/[id]/stats` est la seule route de l'application sans tests d'intégration. Elle contient de la logique métier directement dans le handler (construction de la matchup matrix, calcul des win rates, agrégation de l'ink distribution), ce qui la rend plus risquée qu'une route CRUD standard en cas de régression.

### Critères d'acceptation

- [ ] Créer `src/__tests__/tournament-stats.test.ts`
- [ ] Test : retourne 401 si non authentifié
- [ ] Test : retourne 403 si `groupId` fourni et utilisateur non-membre
- [ ] Test : retourne `scoutingProgress` correct (`fullyScouted / partiallyScouted / unscouted`) à partir de decks connus
- [ ] Test : `inkDistribution` liste les bicolorités avec les bons counts
- [ ] Test : `matchupMatrix.entries` contient les win rates corrects pour des matchs dont les deux bicolorités sont connues
- [ ] Test : retourne des stats vides pour un tournoi sans données
- [ ] Test : retourne 400 pour un ID de tournoi invalide

**Dépendances :** Optionnellement après REFACTO-01 (si le hook est extrait, le test de la route reste identique)
