### SEC-01 — Protéger `DELETE /api/tournaments/[id]` contre toute suppression non autorisée

**Source :** Rapport audit-dette-2026-06-01.md — Section 1.1  
**Priorité :** 🔴 Critique  
**Effort :** S

**Contexte :**  
La route `DELETE /api/tournaments/[id]` ne contient aucun appel à `getAuthSession()`. N'importe quel utilisateur connecté peut supprimer n'importe quel tournoi, sans vérification de rôle. De plus, le middleware autorise les invités (cookie `guest_session`) sur tout `/api/tournaments/*`, rendant la suppression accessible à un guest. Aucun audit log n'est émis lors de la suppression.

**Critères d'acceptation :**
- [ ] La route appelle `getAuthSession(req)` et retourne 401 si absent
- [ ] Seuls les ADMIN et SUPERUSER peuvent supprimer (retourne 403 pour un USER)
- [ ] Un `AuditLogRepository.create` est émis à chaque suppression réussie avec `action: 'TOURNAMENT_DELETE'`
- [ ] Le test existant `'supprime le tournoi et retourne 200'` est mis à jour pour inclure un cookie ADMIN
- [ ] Un nouveau test vérifie que la suppression retourne 403 avec un cookie USER
- [ ] Un nouveau test vérifie que la suppression retourne 401 sans cookie

**Dépendances :** Aucune

### SEC-02 — Supprimer ou protéger `POST /api/rounds/[roundId]/matchs/[matchId]`

**Source :** Rapport audit-dette-2026-06-01.md — Section 1.2  
**Priorité :** 🔴 Critique  
**Effort :** XS

**Contexte :**  
La méthode POST de la route `matchs/[matchId]` upserte directement des données de ronde via `RoundRepository.upsert({ id, ...body })` sans aucune auth ni validation du body. Elle est accessible aux invités via le middleware (GUEST_ALLOWED_API_PREFIXES inclut `/api/rounds/`). Aucun composant frontend ne l'appelle — c'est vraisemblablement du code vestigial. La documentation CLAUDE.md décrit cette route comme `GET/PUT` (pas GET/POST), renforçant l'hypothèse de vestage.

**Critères d'acceptation :**
- [ ] Si supprimée : la méthode POST est retirée du fichier — seul le `GET` reste
- [ ] Si supprimée : vérifier par `grep -rn` qu'aucun composant, hook ou test ne l'appelle
- [ ] Si conservée (scénario alternatif) : protégée par `requireAdminSession`, body validé, test ajouté
- [ ] La suppression n'introduit aucune régression dans `npm test`

**Dépendances :** Aucune

### SEC-03 — Protéger `POST /api/tournaments` contre l'upsert non autorisé

**Source :** Rapport audit-dette-2026-06-01.md — Section 1.3  
**Priorité :** 🟠 Important  
**Effort :** XS

**Contexte :**  
La route `POST /api/tournaments` appelle `TournamentRepository.upsert(body)` sans vérification d'identité ni validation du body. N'importe quel utilisateur connecté peut injecter ou écraser des données de tournoi brutes. La route canonique pour créer un tournoi est `POST /api/tournaments/fetch` (via `TournamentService.fetchAndSave`, avec auth). Cette route POST directe est vraisemblablement vestigiale.

**Critères d'acceptation :**
- [ ] Si supprimée : la méthode POST est retirée — seul le `GET` reste
- [ ] Si conservée (usage interne avéré) : protégée par `requireAdminSession`, body validé contre un schéma dans `validation.ts`
- [ ] Le test existant `'upsert tournament via POST'` est mis à jour en conséquence
- [ ] `npm test` passe à 100%

**Dépendances :** Aucune

### ARCH-01 — Supprimer les `connectToMongoDB()` redondants dans les 5 nouvelles routes

**Source :** Rapport audit-dette-2026-06-01.md — Section 2.1  
**Priorité :** 🟠 Important  
**Effort :** XS

**Contexte :**  
Cinq routes créées lors des features `api-externe-tokens` et `guest-access` appellent `connectToMongoDB()` directement avant d'appeler des repositories. Ce pattern a été explicitement corrigé dans `fix(sec-01)` pour `fetchRound` et `fetchTournament`, mais réintroduit. Chaque repository (`ApiTokenRepository`, `GroupRepository`, `TournamentExternalAccessRepository`) gère lui-même sa connexion en interne — les appels dans les routes sont redondants et violent le principe d'encapsulation des repositories.

**Fichiers concernés :**
- `app/api/external/tournaments/[id]/players/route.ts:2,19`
- `app/api/groups/[id]/tournaments/[tid]/api-tokens/route.ts:6,18,42`
- `app/api/groups/[id]/tournaments/[tid]/api-tokens/[tokenId]/route.ts:6,18`
- `app/api/guest/validate/route.ts:2,18`

**Critères d'acceptation :**
- [ ] Les 5 fichiers n'importent plus `connectToMongoDB` depuis `@/src/lib/db`
- [ ] Les 5 fichiers ne contiennent plus d'appel `await connectToMongoDB()`
- [ ] `npm test` passe à 100% (aucun comportement changé — idempotence du singleton)

**Dépendances :** Aucune

**Critères d'acceptation (useTournamentManagement) :**

### REFACTO-02 — Créer un hook `useTournamentContext` pour centraliser les fetch de `Tournament.tsx`

**Source :** Rapport audit-dette-2026-06-01.md — Section 3.1  
**Priorité :** 🟡 Mineur  
**Effort :** L

**Contexte :**  
`Tournament.tsx` (363 lignes) contient 8 appels `fetch()` directs dans des `useEffect` séparés : `last-round`, `auth/me`, `my-role`, `merge-status`, `conflicts` (user), `conflicts` (groupe), `uncertainties`, et un POST `merge`. Ce composant a grandi d'un fetch à chaque nouvelle feature. Le pattern a déjà été appliqué avec succès pour `StatsTab.tsx` → `useTournamentStats`.

**Critères d'acceptation :**
- [ ] Un hook `useTournamentContext(tournamentId: number, groupId?: string | null)` est créé dans `src/hooks/`
- [ ] Il encapsule les 8 fetch et expose : `{ roundId, currentUserId, groupRole, appRole, hasPendingMerge, conflicts, adminConflicts, uncertainties, refresh }`
- [ ] `Tournament.tsx` descend sous 250 lignes après extraction
- [ ] Tests unitaires pour `useTournamentContext` : cas sans groupId, cas avec groupId (appels conditionnels)
- [ ] `npm test` passe à 100%

**Dépendances :** TEST-02 (pour cohérence du pattern de tests de hooks)
