## Stories de sécurité

### SEC-01 — Aligner validateRegisterBody sur les contraintes du flux guest

**Identifiant :** SEC-01  
**Contexte :** `validateRegisterBody` (utilisé dans le flux d'inscription par invitation) ne valide pas le jeu de caractères du pseudo et ne lowercaseifie pas, contrairement au flux guest (`^[a-z0-9_-]+$` + `.toLowerCase()`). Un utilisateur invité peut créer un compte avec des espaces ou caractères spéciaux dans son pseudo.  
**Criticité :** 🟠 Important  
**Effort :** XS  
**Dépendances :** Aucune  
**Source :** Rapport

**Critères d'acceptation :**
- [ ] `validateRegisterBody` valide le format avec `^[a-z0-9_-]+$`
- [ ] `validateRegisterBody` lowercaseifie le username avant de le retourner
- [ ] Le message d'erreur est cohérent avec celui du flux guest
- [ ] `npm test` passe à 100%
- [ ] Les tests `register.test.ts` couvrent le cas username avec espaces → 400

### TEST-01 — Tests pour GET /api/tournaments/[id]/member-groups

**Identifiant :** TEST-01  
**Contexte :** Route créée pour la feature group-tournament-banner. Aucun test. Logique de membership conditionnelle : on itère sur tous les groupes du tournoi pour filtrer ceux où l'utilisateur est membre.  
**Criticité :** 🟠 Important  
**Effort :** S  
**Dépendances :** Aucune  
**Source :** Rapport

**Critères d'acceptation :**
- [ ] Utilisateur membre de 2 groupes contenant le tournoi → retourne `{ groups: [{groupId, groupName}, {groupId, groupName}] }`
- [ ] Utilisateur non membre d'aucun groupe → retourne `{ groups: [] }`

### TEST-02 — Tests scoutingFilter multi-valeur dans GET /rounds/[roundId]/matchs

**Identifiant :** TEST-02  
**Contexte :** La feature scouting-filter-multiselect a ajouté un paramètre `scoutingFilter` multi-valeur à la route de matchs paginés. `round-paginated.test.ts` couvre déjà `excludeOnePlayer` et la recherche textuelle, mais aucun cas ne valide le filtrage par statut d'assignation.  
**Criticité :** 🟠 Important  
**Effort :** S  
**Dépendances :** Aucune  
**Source :** Rapport couverture

**Critères d'acceptation :**
- [ ] `scoutingFilter=full` → ne retourne que les matchs où les deux joueurs ont des decks complets (2 encres)
- [ ] `scoutingFilter=partial` → ne retourne que les matchs où au moins un joueur a 1 encre assignée
- [ ] `scoutingFilter=none` → ne retourne que les matchs sans deck assigné
- [ ] Combinaison `scoutingFilter=full&scoutingFilter=partial` → union des deux
- [ ] Filtre vide → tous les matchs (comportement actuel inchangé)

### TEST-03 — Tests usePlayerHistory hook

**Identifiant :** TEST-03  
**Contexte :** `usePlayerHistory` n'a pas de test unitaire alors que les autres hooks métier (`useDeckAssignment`, `useMatchState`, `useRound`, `useTournament`) en ont tous. Ce hook effectue une fetch conditionnelle basée sur `playerId` et construit l'URL avec un `groupId` optionnel.  
**Criticité :** 🟡 Mineur  
**Effort :** S  
**Dépendances :** Aucune  
**Source :** Rapport couverture

**Critères d'acceptation :**
- [ ] `playerId = null` → `entries = []`, pas de fetch
- [ ] Fetch réussi → `entries` populées, `loading = false`, `error = null`
- [ ] Fetch échoué → `error` non null, `loading = false`
