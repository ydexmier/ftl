## Liste priorisée des tests à écrire

### Lot 1 🔴 — Prioritaire

#### TEST-01 : `/api/tournaments/[id]/stats` (GET)

Seule route sans test. Contient de la logique métier (calcul matchup matrix, win rates, ink distribution) directement dans le handler.

Cas à couvrir :
- [ ] Retourne 401 si non authentifié
- [ ] Retourne 403 si `groupId` fourni et utilisateur non-membre du groupe
- [ ] Retourne des stats vides (`scoutingProgress.total = 0`) pour un tournoi sans données
- [ ] Retourne `scoutingProgress` correct : `fullyScouted` / `partiallyScouted` / `unscouted` selon les decks assignés
- [ ] Retourne `inkDistribution` avec les bonnes bicolorités et counts
- [ ] Retourne `matchupMatrix.entries` correct après match entre deux joueurs avec bicolorités connues
- [ ] Retourne 400 pour un tournoi ID invalide (NaN)

### Lot 2 🟠 — Important

#### TEST-02 : `useTournament` hook

Ce hook est le point d'entrée principal des données de tournoi côté client. Cas à couvrir :
- [ ] Fetch initial retourne le tournoi
- [ ] `refreshTournament()` recharge depuis l'API
- [ ] Gestion de l'état `loading` et `error`

#### TEST-03 : `useDeckAssignment` hook

Gère l'état du formulaire d'assignation de deck. Cas à couvrir :
- [ ] Initialisation de l'état avec les decks existants
- [ ] Mise à jour d'un deck par matchId
- [ ] Reset de l'état après soumission

### Lot 3 🟡 — Mineur

#### TEST-04 : Chart builders ECharts (si externalisés)

Si `buildDonutOption()`, `buildBarOption()`, `buildHeatmapOption()` sont extraits vers `src/lib/charts/tournamentChartOptions.ts` lors du découpage de `StatsTab.tsx`, les tests unitaires deviendraient trivials à écrire.

Cas à couvrir pour `buildHeatmapOption` :
- [ ] Matrice vide → options sans data
- [ ] Symétrie de la matrice (A vs B = 100 - B vs A pour wins normaux)
- [ ] Cases diagonales → null ou valeur neutre

#### TEST-05 : `useFetch` hook (trivial)

Couvre le hook générique GET. Bas risque, effort minimal. Non prioritaire.
**Suite de tests :** Vitest v4 + MongoDB Memory Server (51 fichiers de test)

## Résumé exécutif

La couverture est passée de 47 à 51 fichiers de test depuis l'audit du 2026-05-19. Les 4 lacunes critiques du lot 1 précédent (`last-round`, `comment-counts`) et les 2 lacunes du lot 2 (`useRound`, `useMatchState`) sont désormais couvertes. La librairie `validation.ts` dispose enfin de tests unitaires dédiés. La nouvelle fonctionnalité inscriptions-tournoi (`registrations`) est arrivée avec son fichier de test. Un nouveau hook complexe `useTournamentManagement` et une nouvelle route `/api/tournaments/[id]/stats` sont arrivés sans tests.

**Alerte sécurité :** trois routes admin (`POST /api/admin/fetchTournament`, `POST /api/admin/fetchRound`, `DELETE /api/admin/tournaments`) n'appellent pas `getAuthSession` / `getAdminSession`. La protection role `ADMIN` du middleware Next.js ne couvre que `/admin/**` (UI), pas `/api/admin/**` (API) — tout utilisateur authentifié (rôle USER) peut déclencher ces opérations. Ce point est détaillé dans la section dédiée.

## 12. Alerte sécurité — Routes admin sans auth serveur

Les trois routes suivantes n'appellent **ni `getAuthSession` ni `getAdminSession`** dans leur code :

| Route | Risque |
|-------|--------|
| `POST /api/admin/fetchTournament` | Tout user authentifié peut déclencher un fetch Ravensburger |
| `POST /api/admin/fetchRound` | Tout user authentifié peut déclencher un fetch ronde |
| `DELETE /api/admin/tournaments` | Tout user authentifié peut supprimer un tournoi en cascade |

**Pourquoi c'est un problème :** le middleware Next.js vérifie le rôle `ADMIN` uniquement pour les routes qui commencent par `/admin` (pages UI). Les routes API `/api/admin/**` ne bénéficient que de la vérification HMAC du cookie — pas du contrôle de rôle. Tout utilisateur avec un cookie de session valide (rôle USER) peut appeler ces endpoints directement.

**Correction recommandée :** ajouter `getAdminSession(request)` en début de chaque handler, retourner `ApiResponse.unauthorized()` si null.
