### ARCH-01 — Supprimer les requêtes Mongoose directes dans les routes auth/user

**Contexte :**  
5 routes importent directement les modèles Mongoose au lieu de passer par les repositories. C'est la violation la plus systémique : si un schéma évolue, les routes ne bénéficient pas des abstractions du repository et peuvent diverger silencieusement.

**Fichiers concernés :**
- `app/api/auth/login/route.ts` — `UserModel.findOne({ username })`
- `app/api/auth/forgot-password/route.ts` — `UserModel.findOne`, `PasswordResetModel.*` (4 requêtes)
- `app/api/user/profile/route.ts` — `UserModel.findOne`, `UserModel.findByIdAndUpdate` (3 requêtes)
- `app/api/user/password/route.ts` — `UserModel.findById`, `user.save()`
- `app/api/feedback/route.ts` — `UserModel.findById` pour récupérer le username

**Critères d'acceptation :**
- [ ] `login/route.ts` utilise `UserRepository.findByUsername()`
- [ ] `forgot-password/route.ts` utilise `UserRepository.findByEmail()` et `PasswordResetRepository` (méthodes create + invalidatePrevious)
- [ ] `user/profile/route.ts` utilise `UserRepository.existsByUsernameExcluding()`, `existsByEmailExcluding()` et `update()`
- [ ] `user/password/route.ts` utilise `UserRepository.findById()` et `UserRepository.updatePassword()`
- [ ] `feedback/route.ts` utilise `UserRepository.findById()`
- [ ] Aucun import de modèle Mongoose dans ces 5 fichiers après la correction
- [ ] Tous les tests existants passent

**Effort :** M  
**Dépendances :** vérifier que `UserRepository` expose bien `findByUsername`, `findByEmail`, `existsByUsernameExcluding`, `existsByEmailExcluding` — les ajouter si absent

### ARCH-02 — Remplacer l'auth manuelle par `getAuthSession()` dans `me` et `admin/groups`

**Contexte :**  
`app/api/auth/me/route.ts` et `app/api/admin/groups/route.ts` réimplémentent l'authentification en 3-4 étapes manuelles au lieu d'appeler `getAuthSession(request)`. Si la logique de validation de session évolue dans `getAuthSession`, ces routes ne bénéficieront pas de la mise à jour — risque de dérive sécurité.

**Fichiers concernés :**
- `app/api/auth/me/route.ts` — auth manuelle + `UserModel` direct
- `app/api/admin/groups/route.ts` — auth manuelle + `GroupModel` direct

**Critères d'acceptation :**
- [ ] `me/route.ts` appelle `getAuthSession(request)` et `UserRepository.findById()`
- [ ] `admin/groups/route.ts` appelle `getAuthSession(request)` avec vérification de rôle ADMIN via `hasRole()`
- [ ] `admin/groups/route.ts` utilise `GroupRepository` au lieu de `GroupModel.find()` — créer `findAll()` si absent
- [ ] Aucun import de `verifyCookie`, `getSession` dans ces fichiers
- [ ] Tous les tests existants passent

**Effort :** S  
**Dépendances :** ARCH-01 (même revue de fichiers auth)

### TEST-01 — Tests d'intégration pour `/api/auth/refresh`

**Contexte :**  
`/api/auth/refresh` est la route centrale du sliding window de session : `SessionGuard` la poll toutes les 4 minutes, elle renouvelle le cookie ET la session en BDD simultanément. Une régression ici provoque des déconnexions silencieuses sans aucune alerte. C'est le seul angle mort 🔴 restant côté sécurité.

**Fichier à créer :** `src/__tests__/auth-refresh.test.ts`

**Critères d'acceptation :**
- [ ] `POST /api/auth/refresh` avec session valide → 200, cookie `Set-Cookie` émis avec `maxAge` de 4h fraîche
- [ ] `POST /api/auth/refresh` avec session valide → `lastActivityAt` mis à jour en BDD
- [ ] `POST /api/auth/refresh` sans cookie → 401
- [ ] `POST /api/auth/refresh` avec cookie dont la signature HMAC est invalide → 401
- [ ] `POST /api/auth/refresh` avec session expirée en BDD (TTL absolu dépassé) → 401
- [ ] `POST /api/auth/refresh` avec session dont `lastActivityAt` dépasse la fenêtre d'inactivité → 401
- [ ] Tous les tests passent avec `npm test`

**Effort :** M

### SEC-01 — Ajouter le rate limiting sur les endpoints publics mutables

**Contexte :**  
Le rate limiting existe uniquement sur `/api/auth/login` (5 tentatives / 15 min par IP). Les deux autres endpoints publics exposés à Internet sans authentification — `POST /api/feedback` et `POST /api/access-requests` — n'ont aucune protection. Ils sont protégés par hCaptcha, mais hCaptcha est contournable et ne remplace pas une limite côté serveur.

**Critères d'acceptation :**
- [ ] `POST /api/access-requests` : rate limiting appliqué par IP (ex. 5 requêtes / 15 min)
- [ ] `POST /api/feedback` : rate limiting appliqué par IP ou par userId (si connecté)
- [ ] La limite est configurable via constante (pas hardcodée)
- [ ] Réponse 429 avec header `Retry-After` si limite dépassée
- [ ] Tous les tests existants passent ; ajouter au moins un test vérifiant le 429 sur chaque route

**Effort :** S  
**Dépendances :** `src/lib/auth/rateLimit.ts` existe — vérifier qu'il est générique ou l'adapter

### VALID-01 — Centraliser la validation des inputs des routes API

**Contexte :**  
Aucune route API ne dispose de schéma de validation déclaratif. Chaque route valide ses entrées manuellement, field by field, avec des conditions `if (!field)` dispersées. Cela entraîne des incohérences (certaines routes valident la longueur, d'autres non), rend les erreurs de validation difficiles à tester exhaustivement, et multiplie le code boilerplate.

**Note :** Ce point est absent des rapports d'audit automatiques mais constitue une dette transversale observable dans l'ensemble des routes API.

**Critères d'acceptation :**
- [ ] Choix d'une solution de validation documenté (Zod recommandé — déjà dans l'écosystème Next.js, compatible TypeScript strict)
- [ ] Les routes 🔴 critiques sont migrées en premier : `login`, `register`, `forgot-password`, `reset-password`
- [ ] Les routes publiques (`/api/access-requests`, `/api/feedback`) sont migrées
- [ ] Chaque schéma couvre : présence des champs obligatoires, types, longueurs max, formats (email, etc.)
- [ ] Les erreurs de validation retournent 400 avec un message structuré via `ApiResponse`
- [ ] Aucune regression sur les tests existants

**Effort :** L  
**Dépendances :** Peut être fait en parallèle des autres stories, mais prioriser les routes ARCH-01/ARCH-02 d'abord
