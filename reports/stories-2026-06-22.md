## SEC-01 — Révoquer les sessions actives lors d'un changement de rôle

**Catégorie :** Sécurité
**Priorité :** 🔴 Critique
**Effort :** S
**Dépendances :** aucune
**Source :** Rapport

### Contexte

Le rôle d'un utilisateur est figé dans son cookie signé au moment du login (`signCookie(sessionId, role)`) et n'est jamais relu depuis le document `User` à jour — ni dans `middleware.ts` (protection des pages `/admin/**`), ni dans `getAuthSession()` (protection des routes API). Quand un admin rétrograde un utilisateur via `PATCH /api/admin/users/[id]`, ses sessions existantes ne sont pas révoquées : il conserve ses anciens privilèges jusqu'à expiration de sa session (jusqu'à 12h) ou révocation manuelle via `DELETE /api/admin/users/[id]/sessions`. C'est un défaut de conception RBAC qui rend une rétrogradation de rôle inopérante en pratique pendant plusieurs heures.

### Critères d'acceptation

- [ ] `PATCH /api/admin/users/[id]` (`app/api/admin/users/[id]/route.ts`) détecte si `role !== undefined && role !== user.role`
- [ ] Si le rôle change, `SessionRepository.deleteByUserId(id)` est appelé après la mise à jour réussie de l'utilisateur
- [ ] Un test d'intégration vérifie : créer une session ADMIN active, la rétrograder en USER via l'API, vérifier que l'ancien cookie ne donne plus accès à une route `requireAdminSession` (401 ou 403 selon le comportement de session expirée)
- [ ] Un test d'intégration vérifie qu'une mise à jour de `username`/`email`/`canCreateGroup` (sans changement de `role`) ne révoque PAS les sessions actives (pas de régression sur le comportement existant)
- [ ] `npm test` passe à 100%

## SEC-02 — Exclure `sessionId` brut de la réponse `/api/admin/users/[id]/sessions`

**Catégorie :** Sécurité
**Priorité :** 🟠 Important
**Effort :** XS
**Dépendances :** aucune
**Source :** Rapport

### Contexte

`SessionRepository.findByUserId` (`src/repositories/db/SessionRepository.ts:10-13`) retourne les documents `Session` complets sans projection, incluant le champ `sessionId` — l'identifiant de session brut. Il est ensuite renvoyé tel quel par `GET /api/admin/users/[id]/sessions`. Ce champ n'est pas affiché dans l'UI admin (qui montre `ipAddress`, `userAgent`, `lastActivityAt`) et constitue une exposition de données inutile (principe de minimisation), même si une prise de contrôle de session directe n'est pas possible sans connaître `SESSION_SECRET`.

### Critères d'acceptation

- [ ] `SessionRepository.findByUserId` exclut le champ `sessionId` via `.select('-sessionId')`
- [ ] La réponse JSON de `GET /api/admin/users/[id]/sessions` ne contient plus le champ `sessionId`
- [ ] Un test d'intégration vérifie que la réponse de cette route ne contient pas la clé `sessionId` dans les objets `sessions[]`
- [ ] `npm test` passe à 100%

## ARCH-02 — Migrer les appels GET directs vers `useFetch`

**Catégorie :** Architecture
**Priorité :** 🟠 Important
**Effort :** M
**Dépendances :** aucune
**Source :** ⚠️ Manquant (identifié hors des rapports précédents — nouvelle découverte du 06-22)

### Contexte

Plusieurs composants font des appels `fetch()` directs en GET au lieu de passer par `useFetch<T>(url)`, le hook standard documenté dans CLAUDE.md : `GroupDetailClient.tsx` (GET `/api/admin/groups/pinned`, ligne 139), `GroupTournaments.tsx` (GET conflicts/uncertainties, lignes 100/107), `GuestInviteModal.tsx` (GET, lignes 43/52), `PlayerCommentHistory.tsx` (GET, ligne 85). Cela contourne la gestion uniforme de loading/error fournie par le hook et duplique cette logique dans chaque composant.

### Critères d'acceptation

- [ ] Les 4 appels GET listés ci-dessus sont remplacés par `useFetch<T>(url)`
- [ ] Le comportement (loading, erreur, données affichées) est inchangé du point de vue utilisateur
- [ ] Les composants concernés n'ont plus de `useState` dédié au loading/erreur de ces appels GET (délégué au hook)
- [ ] Pas de régression sur les tests E2E touchant ces composants (`e2e/tests/groups/`, `e2e/tests/guest/`)
- [ ] `npm test` passe à 100%
